# Designing sitrep.watch: a real-time OSINT conflict monitoring dashboard

**The optimal design for sitrep.watch is a single dark-themed, map-dominant interface with three progressive complexity modes — not three separate apps — built on MapLibre GL JS + deck.gl with a SvelteKit frontend and SSE streaming from the Rust backend.** This architecture mirrors what works at FlightRadar24, Liveuamap, and the Johns Hopkins COVID dashboard while addressing their documented failures: clustering collapse at mid-zoom, alert fatigue at scale, and the "Christmas tree" problem of undifferentiated visual noise. The design must treat 500 events/minute as a data-reduction problem, not a rendering problem — the core UX challenge is surfacing the 3–5 situations that matter right now from a firehose of 720K daily events.

The research below synthesizes patterns from 30+ existing platforms (Liveuamap, ACLED, FlightRadar24, Windy, Bloomberg Terminal, Palantir Gotham, NASA OpenMCT, the Johns Hopkins COVID dashboard), academic literature (Shneiderman's visual information-seeking mantra, Pirolli & Card's sensemaking loop, military Common Operating Picture doctrine), and practical frontend engineering benchmarks into actionable specifications.

## The map-dominant layout with three companion panels

Every successful real-time geospatial dashboard — FlightRadar24, Liveuamap, Global Forest Watch, USGS Earthquake Monitor, Windy — dedicates **70–80% of screen real estate to the map**. The remaining space hosts companion panels that transform the map from a visualization into an analytical workspace. The critical lesson from Liveuamap (the closest existing analog to sitrep.watch) is that a map + scrollable event feed is the minimum viable layout, but the JHU COVID dashboard proved that map + summary statistics + trend sparklines creates something universally understood.

The recommended layout uses four zones. A **thin persistent header** (48px) carries the mode selector (Simple | Research | Analyst), global search, notification bell with unread count, and a real-time activity sparkline showing aggregate event rate across all sources. A **collapsible left sidebar** (320–480px, reducible to a 48px icon rail) holds the situation list as expandable cards, source layer toggles grouped into six semantic themes, and severity/time filters. The **map canvas** fills all remaining space with MapLibre rendering the basemap and deck.gl rendering all data layers. A **resizable bottom sheet** (three snap positions: collapsed toolbar, 33% height, 66% height) hosts the timeline view, event detail panels, and raw data tables for analyst mode.

This layout directly mirrors Bloomberg Terminal's Launchpad philosophy of user-composable panels. Bloomberg's UX research found that expert users resist simplification and take pride in handling complex interfaces — validating the approach of density for analysts with progressive simplification for casual users rather than dumbing down the default. NASA's OpenMCT mission control framework uses the same composable pattern: drag-and-drop data objects into customizable layouts, with pre-composed screens for standard operations and full customization for specialized analysis.

On mobile, the sidebar collapses entirely. The bottom sheet becomes the primary interaction surface following the Google Maps mobile pattern: tap a marker, bottom sheet slides up with situation summary, swipe up for full detail. Mode defaults to Simple on mobile screens under 768px.

## Three audience tiers through a single progressive interface

The strongest finding across the research is that **separate interfaces for different user tiers are a maintenance trap and a UX anti-pattern**. Nielsen Norman Group research explicitly warns against this approach. Instead, sitrep.watch should implement Shneiderman's mantra — "overview first, zoom and filter, then details on demand" — through a three-position mode toggle that controls information density within the same underlying interface.

**Simple mode** (general public) shows only high-severity active situations as narrative cards in the sidebar, each with a plain-language headline, location, timestamp, and a two-sentence summary. The map displays situation polygons (convex hulls around event clusters) colored by severity, with count badges. No source toggles, no raw data, no filtering beyond clicking a situation to read its story. This mirrors how Al Jazeera's AJLabs presents conflict data: story-driven rather than dashboard-driven, with each map element answering a specific question. Typography should feel editorial — Inter at 16px body, generous line-height of 1.5, thumbnail images where available.

**Research mode** (journalists and investigators) adds source attribution per event, confidence indicators, a horizontal timeline view per situation, temporal range selectors, and export capabilities (PDF situation report, CSV data, KML geographic data, embeddable widget code, shareable permalink). Each situation card expands to show constituent events with individual source citations. The timeline becomes a key analytical tool — a horizontal scrollable axis showing event progression within a situation, with each node linking to source material. This tier draws from ACLED's Explorer pattern: extensive filters by location, actor type, event type, and date range, combined with Bellingcat's emphasis on source provenance and verification methodology.

**Analyst mode** (OSINT researchers) exposes maximum density. All 26 source toggles appear in the sidebar grouped by theme. A multi-panel layout becomes available: map + feed + timeline + raw data table visible simultaneously, with drag-to-resize dividers between panels. Keyboard shortcuts enable rapid navigation (Bloomberg Terminal's keyboard-first interaction model). Custom saved views/workspaces persist across sessions. API access documentation is linked directly from the interface. Source health indicators show freshness/staleness/error status for each of the 26 data sources — a pattern pioneered by the open-source WorldMonitor dashboard that explicitly reports "intelligence gaps" (what analysts *cannot* see). This acknowledges that **knowing which sources are down is as important as the data itself**.

Mode selection persists via localStorage or user account, with URL parameter support (`?mode=analyst`) for shareable mode-specific links. The transition between modes should animate smoothly, showing elements appearing or disappearing over 300ms to maintain spatial memory.

## Visual encoding: severity, confidence, recency, and the anti-Christmas-tree system

The single most common failure mode in OSINT dashboards is the "Christmas tree" problem: every data point gets a unique color, size, and shape, and nothing stands out. The antidote is a disciplined encoding system where **color encodes severity, size encodes magnitude, opacity encodes recency, and border style encodes confidence** — each visual channel maps to exactly one data dimension.

**Severity uses a warm sequential palette** that avoids red-green confusion and works for all color vision deficiency types: Critical `#DC267F` (magenta), High `#FE6100` (safety orange), Medium `#FFB000` (selective yellow), Low `#648FFF` (ultramarine blue), Info `#48484A` (dark gray). This IBM colorblind-safe palette was validated against protanopia, deuteranopia, and tritanopia simulations. The deliberate choice to make Critical magenta rather than red prevents the "everything is red" problem that plagues conflict maps, and reserves true red (`#F85149`) exclusively for system-level critical alerts (data source failure, authentication errors) so it remains a genuine pattern-break.

**Temporal recency uses opacity decay**: events under 5 minutes old render at full opacity with a 2-second CSS pulsing glow animation; 5–60 minutes at 100% opacity without animation; 1–24 hours at 80%; 1–7 days at 50%; 7–30 days at 25% with minimum marker size. The USGS earthquake map proved this double-encoding strategy (color for one dimension, opacity for another) creates immediate visual urgency without user training.

**Confidence uses border treatment**, borrowing from the NATO Admiralty Code's two-axis system (source reliability × information credibility). High-confidence events (3+ corroborating sources) get a solid 2px border. Medium confidence (2 sources) gets a solid 1px border. Low confidence (single unverified source) gets a dashed 1px border with reduced saturation. A small ring indicator around each marker fills proportionally to source diversity — visible proof of corroboration at a glance.

**Iconography should be simplified civilian symbology**, not raw MIL-STD-2525. Military symbols require trained operators — research from DTIC shows even trained users struggle to distinguish hostile from suspect affiliations when symbols overlap, and icon presence significantly increases visual search time. Instead, create a compact SVG icon set inspired by OCHA humanitarian icons: crossed swords for armed clashes, burst star for explosions, concentric circles for shelling, raised fist for protests, flame for fires/thermal anomalies, broken building for infrastructure damage, anchor for maritime events. The geometric frame shape from MIL-STD-2525 adapts elegantly for civilian use: **circles for ongoing situations, diamonds for unconfirmed/single-source events, squares for confirmed multi-source events** — encoding verification status through shape without requiring military training.

## Temporal visualization: timelines, velocity, and the 30-day window

Conflict monitoring demands temporal awareness at four scales simultaneously: what is happening right now (seconds), what has developed today (hours), what is the trend this week (days), and what is the broader trajectory (weeks). No single temporal widget serves all four.

The **activity sparkline** in the header bar provides ambient awareness of global event velocity — a compact line chart showing aggregate events per hour for the past 24 hours. This draws from Bloomberg's approach to temporal density: a vital sign always visible, conveying system health at a glance. When the sparkline spikes, something is happening somewhere.

The **situation timeline** in the bottom sheet panel serves Research and Analyst modes. A horizontal scrollable axis with time on X, event markers colored by type and sized by severity, and a vertical "now" line with live events streaming in from the right edge. Brush selection on the timeline cross-filters the map, showing only events within the selected time window. ACLED's Trendfinder demonstrates the value of overlaying a 4-week moving average trendline with anomaly highlighting (2+ standard deviations above baseline flagged as "High Activity Anomaly"). The GDELT Timeline Visualizer shows a brilliant compact pattern: years on Y-axis, days on X-axis, with semi-transparent dots — revealing anniversary patterns, seasonal cycles, and multi-year trends at daily resolution. This view should be available in Analyst mode.

**Situation velocity** — whether an incident is escalating, stable, or de-escalating — is the single most actionable temporal signal for all three user tiers. Calculate a rolling event rate for each geographic cluster over trailing 1h, 4h, and 24h windows. Compare current rate to historical baseline using ACLED's approach. Encode velocity as: escalating (upward arrow, warm pulse, `#FF453A`), stable (horizontal dash, neutral, `#FFD60A`), de-escalating (downward arrow, cool tone, `#30D158`), new/emerging (pulsing flash, `#FFB000`). Embed a micro-sparkline with direction arrow directly in each situation card — the most information-dense way to convey trajectory without requiring the user to open a timeline.

**Time slider with playback** enables temporal exploration: dual-handle range slider, play/pause/step controls, speed selector (1x through 10x). ArcGIS patterns show three essential display modes: "show current" (only events at the current timestamp), "show window" (events within the range), and "show cumulative" (all events up to now). For a 30-day retention window with 21.6M total events, server-side aggregation is mandatory — the client requests pre-aggregated data at the temporal resolution appropriate to the zoom level (hourly aggregates for 30-day view, individual events for hours-level zoom).

## Real-time updates: ambient awareness without alert fatigue

At 500 events per minute, treating each event as individually notable would destroy usability. The core principle from signal detection theory in UX research is that the system must maximize hits (surfacing critical events) and correct rejections (suppressing noise) while minimizing both misses (failing to surface something important) and false alarms (the direct cause of alert fatigue).

The **four-tier notification architecture** handles this. **Flash alerts** (fewer than 1/day) — mass casualty events, confirmed WMD use, major territorial changes — trigger a full-screen overlay banner, auto-pause the dashboard, and optionally play an audio alert. **Breaking alerts** (a few per day) — major battles, significant escalations, high-confidence multi-source events — show as a persistent top banner with auto-pan to location on the map, and a priority card with orange border in the sidebar. **Notable events** (dozens per day) — significant events, trend anomalies, new actor emergence — insert as new sidebar cards with a gentle 300ms slide animation and brief marker glow on the map. **Routine events** (~500/minute) — the vast majority — are batch-aggregated into cluster counts, sparkline updates, and heatmap density refreshes with no individual notification.

**Ambient awareness patterns** keep analysts informed without demanding active attention. A peripheral glow shifts the map border color based on global alert level: normal (`#1C1C1E`) transitions slowly over 30 seconds to elevated (`#3A1C1C`) to critical (`#4A1C1C`) — operating in peripheral vision like weather changing outside a window. Delta indicators (▲/▼ with percentage) on each region card show event rate changes. A narrow event stream waterfall (like a stock ticker) scrolls the most recent events in a semi-transparent column — clicking it pauses and expands for detail.

**Intelligent deduplication** is the most important anti-fatigue mechanism: 50 events about the same airstrike should produce 1 situation card with an event count, not 50 notifications. This is the same pattern SOC (Security Operations Center) dashboards use, where 80–95% of raw alerts are noise. Cooldown timers suppress similar alerts from the same region for a configurable period. An attention budget cap limits active notifications to 3 visible at once. A "Do Not Disturb" mode lets analysts focus on investigation with only Flash events breaking through.

## Technical stack: MapLibre + deck.gl + SvelteKit + SSE

**MapLibre GL JS is the clear primary mapping library** — open-source (BSD-2-Clause), GPU-accelerated WebGL vector tile rendering, service-agnostic (no vendor lock-in), and benchmarked to handle 50K+ features efficiently. Mapbox GL JS v2+ is proprietary and incompatible with an open-source project. Leaflet's DOM-based rendering degrades beyond hundreds of markers. CesiumJS adds 3D complexity without proportional value for 2D event mapping.

**deck.gl v9 serves as the visualization overlay**, rendering all event data as GPU-accelerated layers on top of MapLibre. deck.gl handles 100K–1M+ points trivially through WebGL2, and its `@deck.gl/mapbox` module supports interleaved rendering (sharing MapLibre's WebGL context for proper z-ordering with map labels). Key layers: ScatterplotLayer for event markers, IconLayer for typed event icons, HeatmapLayer for density at low zoom, GeoJsonLayer for situation polygons and boundaries. For real-time data, deck.gl's `_dataDiff` API enables incremental attribute updates instead of full recalculation, and constant accessor props (e.g., `getFillColor: [255,0,0]`) are dramatically cheaper than function accessors.

**SvelteKit (Svelte 5) is recommended over React** for the frontend framework. Svelte 5's fine-grained reactivity compiles away at build time with no virtual DOM diffing overhead — state changes surgically update only affected DOM nodes, ideal for high-frequency real-time updates. React's virtual DOM reconciliation becomes a bottleneck requiring careful optimization (refs for high-frequency data, isolated components, throttled setState). Svelte's smaller runtime bundle (~45KB savings over React + ReactDOM) benefits global users on varying network quality. deck.gl works standalone via vanilla JS API and integrates through Svelte's `onMount` and `$effect`. The `svelte-maplibre-gl` library (by MIERUNE) provides a Svelte 5 native wrapper with full runes system support.

**Server-Sent Events (SSE) should be the primary real-time protocol**, not WebSockets. sitrep.watch is fundamentally a server→client data flow. SSE provides automatic reconnection with `Last-Event-Id` for resumption, works with standard load balancers and CDNs, and HTTP/2 multiplexing eliminates the old 6-connection limit. Performance is identical to WebSocket for server→client streaming. Axum (the recommended Rust web framework) has first-class SSE support via `axum::response::sse`. Use `tokio::sync::broadcast` channels to fan out events to all connected clients. Reserve WebSockets for the one bidirectional need: viewport-aware subscriptions where the client sends viewport bounds and the server responds with only relevant events.

**Protomaps PMTiles on Cloudflare R2 is the ideal tile serving strategy** — a single-file format using HTTP Range Requests to serve tiles from static storage with no tile server needed and zero egress cost on R2. The `pmtiles` npm package integrates with MapLibre via `addProtocol()`. For dynamic event data rendered as vector tiles, **Martin** (a Rust-native tile server from the MapLibre project) generates tiles on-the-fly from PostGIS and serves PMTiles/MBTiles — a perfect fit for sitrep.watch's Rust ecosystem.

**Client-side clustering uses Supercluster** (by Mapbox's Vladimir Agafonkin), which processes 400K points in under 1 second across all zoom levels, with subsequent zoom-level queries completing in under 10ms. A Web Worker architecture offloads SSE parsing, Supercluster indexing, RBush spatial queries, and event filtering from the main thread, communicating via `postMessage` with Transferable ArrayBuffer objects for zero-copy data transfer. The client should maintain a rolling 24–48h window in memory (~720K–1.4M events, ~100–360MB with binary serialization) with older data fetched on demand via API and cached in IndexedDB.

## Professional dark-themed aesthetics that convey authority

The visual language of sitrep.watch must communicate the same institutional credibility as Bloomberg Terminal, BBC News, and NASA mission control — serious, precise, restrained. Research across these reference interfaces reveals consistent principles.

**The background should be dark blue-gray, never pure black.** Pure black (#000000) creates harsh contrast, reduces elevation perception, and causes visual vibration with white text. Material Design, IBM Carbon, and Bloomberg's own research confirm this. The recommended background hierarchy uses four elevation levels: base `#0D1117`, surface-01 `#161B22`, surface-02 `#1C2128`, surface-03 `#242A33`, with subtle borders at `#30363D`. This is GitHub's dark theme palette — proven at massive scale, tested for accessibility, and immediately familiar to the developer/researcher audience.

**Typography uses Inter for UI and JetBrains Mono for data.** Inter was designed specifically for computer screens with excellent legibility at small sizes, variable weight support, and tabular (fixed-width) numbers critical for data alignment. JetBrains Mono provides increased letter height with standard width for optimal data readability. The pairing signals technical credibility without coldness. Text hierarchy on dark backgrounds should use `#E6EDF3` (87% white) for primary text, `#8B949E` (60% white) for secondary labels, and `#484F58` for disabled/tertiary content. A lightweight font weight (400) works better on dark backgrounds because light text appears brighter than its nominal weight.

The type scale should be compact but readable: 28px/600 for dashboard titles, 20px/600 for section headers, 14px/400 for body text (minimum for extended reading), 12px/500 for labels, 11px/400 for timestamps and attribution. All sizes in rem, never px, to respect user browser settings. Line-height minimum of 1.375 for body text following BBC Reith guidelines.

**Spacing uses an 8px base unit** following IBM Carbon's 2x grid system. A 16-column responsive grid accommodates the sidebar + map + bottom sheet layout with condensed 1px borders between tiles for maximum data density in Analyst mode. The F-pattern scanning model places the most critical data (situation summary, alert count, global status) in the top-left quadrant.

**What conveys authority across all reference platforms** is remarkable consistency: 1–2 font families maximum, clear hierarchy through weight and size rather than decoration, restrained color palettes (3–5 colors maximum in any single view), prominent source citations, and visible methodology documentation. Al Jazeera, BBC, and Reuters all achieve trustworthiness through *restraint* — every visual element earns its place, nothing is decorative, and data attribution is never hidden.

## Ethical guardrails and accessibility requirements

A public-facing conflict monitoring dashboard carries ethical obligations beyond standard software. OCHA's Centre for Humanitarian Data establishes four principles: do no harm (data presentation must not endanger affected populations), data minimization (display only what's necessary), purpose limitation, and privacy by design.

**Location precision must default to administrative-level aggregation** — district or governorate level — with drill-down to finer precision available behind a clear interaction barrier. The W3C's Responsible Use of Spatial Data guidelines note that "spatial data may be seen as a fingerprint — every combination of location in space, time, and theme is unique." Events involving civilian casualties should use location fuzzing (5km radius minimum) in default views. Precise coordinates should be available only in Analyst mode with explicit acknowledgment.

**Content should use a three-tier sensitivity system.** Level 1 (default): aggregate counts, summary statistics, map markers with clinical descriptions. Level 2 (detail): full event descriptions in neutral language, available on click. Level 3 (raw): explicit descriptions, raw source material, available only behind an opt-in toggle with interstitial warning. Casualty data must be presented with appropriate gravity — these are people, not metrics. Avoid gamification aesthetics entirely: no "scores," no leaderboards, no celebratory animations.

**WCAG 2.1 AA compliance is the minimum target**, with AAA where achievable. Concrete requirements: 4.5:1 contrast ratio for normal text, 3:1 for large text and graphical UI components, never relying on color alone to convey information (always pair with shape, pattern, icon, or text label), keyboard navigation for all interactive elements including map pan/zoom via arrow keys, visible 2px focus indicators, skip-navigation links for complex layouts, and data tables as an alternative to map visualization for screen reader users. Touch targets must be minimum 44×44px. Content must reflow at 200% zoom without horizontal scrolling.

The accent color palette should avoid red-green pairings entirely (affecting ~8% of male users). The IBM colorblind-safe palette (`#648FFF`, `#785EF0`, `#DC267F`, `#FE6100`, `#FFB000`) is validated across all major CVD types and works on dark backgrounds. The severity scale must use quad encoding — shape + color + size + text label — so no information is lost to any single sensory channel.

## Conclusion: the design decisions that matter most

Three architectural decisions will determine whether sitrep.watch succeeds or becomes another cluttered OSINT map. **First, situations (not individual events) must be the primary unit of interaction.** The system's semantic/temporal/geographic clustering is its core differentiator — the UI must surface these clusters as narrative-ready situation cards with velocity indicators, not just dump 720K daily dots on a map. Every design decision should optimize for situation-level comprehension.

**Second, the mode toggle (Simple/Research/Analyst) must feel like zooming into a microscope, not switching applications.** The same data, the same map, the same situations — just progressively more controls, more sources, more raw detail. This is what the JHU COVID dashboard got right: a nurse and an epidemiologist could both use the same interface at different depths. Store the mode preference, honor URL parameters for sharing, and default intelligently based on device.

**Third, the real-time pipeline must be a data-reduction pipeline, not a data-firehose.** Server-side clustering, intelligent deduplication, tiered notifications with aggressive suppression of routine events, and batch aggregation for the map — these are backend design decisions that determine frontend usability. The hardest UX problem is not rendering 50K markers (deck.gl handles this trivially); it is ensuring that when a situation genuinely escalates, the dashboard makes this unmissable without crying wolf 500 times per minute.

The technical stack of MapLibre GL JS + deck.gl + SvelteKit + SSE + Protomaps/Martin is battle-tested, fully open-source, and optimized for exactly this workload. Start with the map + situation cards + activity sparkline. Add the timeline. Then add the mode toggle. The interface should be useful on day one of development, with each subsequent feature increasing depth rather than complexity.
