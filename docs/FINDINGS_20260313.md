# Findings Report — March 13, 2026

## Executive Summary

Three severity-scoring bugs were identified and fixed that caused 17 HIGH-severity situations where only 3 were legitimate. The root causes were: dwell-time phase blocks skipping severity recomputation, raw source counts inflating diversity (GDELT+GdeltGeo+RSS counted as 3 instead of 1), and overly permissive conflict-path logic. Additionally, two new maritime data sources were deployed (IMB Piracy, UKMTO PDF extraction), and long-running situation memory was implemented to give the AI narrative engine continuity across regeneration cycles.

---

## Severity Calibration

### Bugs Found

Three bugs in `pipeline/src/situation_graph/lifecycle.rs` combined to produce inflated severity scores:

| # | Bug | Root Cause | Impact |
|---|-----|-----------|--------|
| 1 | **Dwell-time recompute skip** | When a Declining phase transition was blocked by dwell time (12h for HIGH, 24h for CRITICAL), the `continue` statement skipped the severity recomputation loop entirely. Clusters loaded from the DB retained their stale HIGH severity indefinitely. | Situations that should have been downgraded to MEDIUM or LOW remained HIGH forever. |
| 2 | **Raw source diversity count** | `recompute_cluster_severity` used `effective_source_types.len()` instead of `effective_source_diversity()`. The latter collapses GDELT + GdeltGeo + RssNews into 1 "news" source, flight trackers into 1 "aviation" source, and GDACS + Copernicus into 1 "disaster" source. | A cluster with GDELT, GdeltGeo, and RssNews was counted as 3 sources, easily meeting the 2+ threshold for HIGH. |
| 3 | **Conflict path too permissive** | The HIGH conflict path accepted `has_conflict_sources OR has_conflict_topics`. Since almost every cluster has GDELT (a conflict source), virtually all met the condition. | Changed to require `has_conflict_topics AND 3+ effective sources` to ensure genuine multi-source corroboration. |

### Fix (commit `d6077f2`)

- Replaced `continue` with a `phase_blocked` flag so severity recomputation always runs.
- Swapped `source_types.len()` for `effective_source_diversity()` in the severity function.
- Changed HIGH conflict-path from `OR` to `AND` with a minimum 3 effective sources.

### Before/After Metrics

| Metric | Before Fix | After Fix |
|--------|-----------|-----------|
| CRITICAL | 1 | 1 |
| HIGH | 17 | 3 |
| MEDIUM | 2 | 12 |
| LOW | 12 | 20 |
| Total situations | 32 | 36 |

Active recomputation was confirmed in production logs: situations like "Brussels Houthi Terror Designation Push" were downgraded from HIGH to MEDIUM on the first sweep cycle after deploy.

## New Sources

### IMB Piracy Reporting Centre (C.2)

**Commit**: `cea1aad`
**Source file**: `sources/src/imb.rs`
**Source ID**: `imb-piracy`
**Poll interval**: 1 hour

The ICC-CCS International Maritime Bureau maintains a Live Piracy Map backed by a WordPress plugin (WP Go Maps). The source polls the REST API at `icc-ccs.org/wp-json/wpgmza/v1/markers` for incident markers spanning 2020-2024 (map IDs 3, 4, 5, 9, 10).

**Data pipeline**:
1. Fetches markers from the most recent map ID that has data.
2. Extracts structured fields from `custom_field_data`: incident number, sitrep text.
3. Parses sitrep for date (`DD.MM.YYYY`), time (`HHMM UTC/LT`), coordinates (`Posn: DD:MM.MN - DDD:MM.ME`), and location name.
4. Applies severity from ICC-CCS categories (1-5) with keyword escalation (hijack/kidnap -> CRITICAL, fired/RPG -> HIGH).
5. Generates tags for maritime regions (Red Sea, Gulf of Aden, Somalia, Nigeria, Indonesia, Singapore) and incident types (armed, robbery, boarding, skiff).
6. Deduplication via `imb:{year}:{incident_number}` source IDs.
7. 30-day lookback to avoid compressed TimescaleDB chunk insertion failures.

**Event types**: `MaritimeSecurity` for general incidents, `ConflictEvent` for hijackings/kidnappings/armed attacks.

## UKMTO PDF Text Extraction

**Commit**: `61f86ac`
**Source file**: `sources/src/ukmto_warnings.rs`

Previously, the UKMTO Warnings source created events solely from PDF filenames scraped from the MSCIO mirror HTML listing. This update adds full PDF content extraction:

1. Downloads the PDF (capped at 5MB).
2. Extracts text using the `pdf-extract` crate.
3. Parses extracted text for:
   - **Warning type**: Attack, Suspicious Approach, Hijack, Boarding, Firing, etc.
   - **Coordinates**: DMS format (`DD-MM.MN / DDD-MM.ME`) from warning body.
   - **Description**: Full warning narrative text.
4. Falls back to metadata-only event creation if PDF download or parsing fails.

The existing coordinate and warning-type parsers were already implemented but were dead code. This commit wires them up to the actual PDF content pipeline.

## Long-Running Situation Memory

**Commit**: `3d24200`
**Migrations**: `017_situation_summaries.sql`, `018_situation_timeline.sql`

### Problem

Situations that run for days or weeks lose context across narrative regenerations. The narrative engine only sees the latest 15 events and the previous narrative text, so it has no memory of early key developments after they age out.

### Solution

Two new database tables and an AI-powered summarization loop:

**`situation_summaries`** (1 row per situation):
| Column | Type | Purpose |
|--------|------|---------|
| `situation_id` | UUID PK | Links to situation |
| `summary_text` | TEXT | 3-5 sentence rolling summary |
| `key_entities` | JSONB | Top 10 entities (people, orgs, locations) |
| `key_dates` | JSONB | Up to 8 milestone date/event pairs |
| `updated_at` | TIMESTAMPTZ | Last summary generation time |

**`situation_timeline`** (hourly buckets per situation):
| Column | Type | Purpose |
|--------|------|---------|
| `situation_id` | UUID | Links to situation |
| `bucket` | TIMESTAMPTZ | Hour-aligned bucket |
| `event_count` | INT | Events in this hour |
| `source_count` | INT | Distinct source types in this hour |
| `max_severity` | TEXT | Peak severity in this hour |

**Summary generation** (`intel/src/narrative.rs`):
- Triggers every 50 new events or 6 hours, whichever comes first.
- First summary requires 10+ events.
- Uses Ollama (Qwen 3.5) first, falls back to Gemini Flash-Lite.
- Produces structured JSON: `summary`, `key_entities[]`, `key_dates[]`.
- The `previous_summary` is injected into the narrative prompt under "Cumulative Summary (situation memory)" so the narrative engine maintains continuity.

**Timeline materialization**: Buckets are materialized every 5 minutes from the events table. Exposed via `GET /api/situations/{id}/timeline`.

## Quality Debt Analysis

### Pre-Fix Assessment (17 HIGH situations)

Before the severity fix, 17 out of 32 situations were rated HIGH. The three bugs described above meant that virtually any cluster with GDELT events (the most common source) could reach HIGH severity because:

1. GDELT is classified as a "conflict source," and the `OR` gate with `has_conflict_sources` was always satisfied.
2. GDELT + GdeltGeo + RssNews counted as 3 raw source types, satisfying the 2+ diversity threshold.
3. Once a situation reached HIGH during its Active phase, dwell-time blocks on the Declining transition also blocked severity recomputation, preventing demotion.

### Post-Fix Distribution

After deploying `d6077f2`:

- **3 remaining HIGHs**: These are situations with genuine multi-source corroboration (3+ effective source types), conflict-relevant topics, and 25+ events. Legitimate.
- **12 MEDIUMs**: Former HIGHs that had either fewer than 3 effective sources or lacked conflict topic keywords. Correctly downgraded.
- **20 LOWs**: Small clusters with limited corroboration.
- **1 CRITICAL**: Unchanged (requires 40+ events, 4+ source types, Active phase, conflict sources + topics).

The fix was validated within 14 minutes of deployment (21:50 to 22:04 UTC). The first sweep tick after restart triggered recomputation across all 36 situations, with log evidence of active downgrading.
